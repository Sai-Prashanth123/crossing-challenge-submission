# CLAUDE.md — Notes on AI-Assisted Development

This submission was built with **Claude Code** (Anthropic's CLI for Claude) using a structured **spec → plan → execute** workflow. Including this file is required by the challenge rules; below is an honest account of how the AI was used and where its leverage was real vs marginal.

## Workflow

1. **Brainstorm** (`docs/superpowers/specs/2026-04-30-crossing-challenge-design.md`)
   Talked through the problem, constraints (CPU laptop + Colab T4 free tier, fast time budget), and three candidate approaches (pure-CPU classical ML, hybrid LGBM + GRU, full transformer). Picked the hybrid for the best ratio of expected score gain to time spent.

2. **Spec self-review**
   Caught two real bugs before writing any code:
   - A test threshold that used the *Eval* baseline (0.74) as a *Dev* regression gate (Dev baseline is 0.83, so the gate would have been useless).
   - A claimed feature count of 30 in the spec that didn't match the array I'd allocated in code (33 floats).

3. **Implementation plan** (`docs/superpowers/plans/2026-04-30-crossing-challenge-implementation.md`)
   13 TDD tasks, each with exact file paths, code blocks, success criteria, and commit messages. The plan was the contract; subagents executed tasks against it one at a time.

4. **Execute**
   Subagent-driven: a fresh Claude subagent per task, isolated context, each one committing its work with a measurable result (e.g., `intent: lightgbm with grouped CV + isotonic, dev BCE 0.21`). Reviewed between tasks; corrected when results missed targets.

## Where Claude Code helped most

- **Scaffolding speed**. The 33-feature `featurize_intent`, the GRU training loop with cosine warmup and best-checkpoint saving, the dataset class, and the test files were drafted in minutes and tightened by hand. Estimated time savings: 4–6 hours.
- **Defensive code generation**. Every sanitization clamp in `predict.py` (`[-2000, 4000]` for bboxes, `[1e-6, 1-1e-6]` for intent, NaN-replacement to frame center) was generated by reading `grade.py` directly. Local Dev score and the real Gobblecube grader should agree to the digit.
- **Diagnostic-driven retraining**. After the first Colab run landed at 34 px ADE (worse than expected), Claude proposed three specific named-hypothesis fixes:
  1. *Heavy-tailed +2.0 s residuals* → switch `MSELoss` to `SmoothL1Loss(beta=0.005)`
  2. *Capacity ceiling* → bump GRU `hidden` from 64 to 128
  3. *Weak ego signal muddied by augmentation* → drop the speed-scaling augmentation
  All three landed correctly; retrained model went from 34 → 28.75 px ADE in one more Colab session.
- **Honest failure surfacing**. When the LightGBM intent BCE missed its 0.20 target and landed at 0.2125, the implementer subagent stopped and reported `BLOCKED` rather than committing — which gave me a chance to relax the gate honestly rather than hide a partial result.

## Where it didn't help

- **The modeling intuition.** Decisions like *predict residuals over constant velocity* (standard PIE/JAAD literature trick), *hold bbox size constant* (residuals-vs-noise tradeoff), and *augment ego-speed for the sidewalk-robot distribution shift* (called out explicitly in the challenge FAQ) came from reading the papers and brief carefully. Claude was a fast pair, not the architect.
- **Numerical / hyperparameter judgement.** The original spec's claim that intent BCE could hit 0.18 turned out optimistic; reality landed at 0.21. The original GRU spec's ADE target of 19 px was also optimistic; landed at 28.75. Claude over-projected because the spec's data on what's achievable came from extrapolation, not from running the experiments. Recalibrated honestly mid-stream.
